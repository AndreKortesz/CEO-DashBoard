## 2. Путь клиента через CRM

### Три воронки Битрикс24:

```
ВОРОНКА ЛИДОВ (crm.lead.*)
─────────────────────────
Новый → Связаться с клиентом → Нужны доп. данные → В работе → Предварительное КП → Качественный лид (УСПЕХ = записан на осмотр)

Причины отказа:
- Не клиент
- Не дозвон
- Неуспешный лид
- Дубль
- Почта
- Отложено на будущее
```

```
ВОРОНКА ВЫЕЗДОВ (crm.deal.*, category_id=45)
─────────────────────────────────────────────
Согласование выезда → Выезд назначен → Выезд проведён, заявка на инженера → Выезд выполнен (УСПЕХ + копия остаётся)
                                                                           → Осмотр не произведён (ОТКАЗ)

Типы выездов (определяются по префиксу в названии):
- О — Осмотр
- М — Монтаж
- Г — Гарантия
- Диагн — Диагностика

ВАЖНО: одна и та же воронка используется для ВСЕХ типов выездов.
При успешном осмотре создаётся сделка в воронке сделок (category_id=7).
```

```
ВОРОНКА СДЕЛОК (crm.deal.*, category_id=7)
──────────────────────────────────────────
Согласование выезда → Осмотр проконтролирован → КП на рассмотрении → КП >2 нед → КП >3 мес → Счет выставлен → Контроль оборудования → Монтаж завис (ЛОВУШКА) → Монтаж произведён → Монтаж проконтролирован → Контроль произведён → Сделка успешна

"Монтаж завис" — специальная стадия-ловушка для проблемных проектов.
"Сделка успешна" — финальная стадия ПОСЛЕ проверки РОПом всех сумм, дат, документов.
```

### Полный путь клиента:
```
Лид (воронка лидов)
  ↓ менеджер записывает на осмотр
Выезд (воронка выездов, cat 45)
  ↓ монтажник выполняет осмотр
Сделка (воронка сделок, cat 7) + копия в выездах
  ↓ менеджер готовит КП, согласование, счёт
  ↓ закупка оборудования
Выезд на монтаж (та же воронка выездов, cat 45)
  ↓ монтажник выполняет монтаж
Контроль + проверка РОП → Сделка успешна
```

## 3. Ключевые конверсии

Дашборд измеряет три точки воронки: **лид → осмотр → монтаж**.

Промежуточные стадии (КП, счёт, контроль) видны в детализации сделок, но для конверсий считаются только эти три.

**Конверсии менеджера:**
- Лид → Осмотр (менеджер записал, монтажник приехал)
- Осмотр → Монтаж (от осмотра до завершённого монтажа)
- Общая: Лид → Монтаж

**Конверсии монтажника:**
- Осмотр назначен → Осмотр проведён
- Монтаж назначен → Монтаж выполнен
- % гарантийных возвратов (монтаж → потом гарантия на тот же объект = проблема качества)

**Конверсии по направлениям** (СКУД, Видео, Умный дом, Усиление связи, Пожарка, Wi-Fi):
- Те же этапы лид→осмотр→монтаж, но в разрезе направления

**Дедупликация**: при подсчёте, если на одну сделку было 2 осмотра — считаем как 1 осмотр. Если 2 монтажа на одну сделку — считаем как 1 монтаж. Ключ дедупликации — deal_id.

### 4.1 Bitrix24 (Фаза 1)
- **Тип**: REST API через входящий вебхук

- **Поле "направление"**: есть в лидах, позволяет разбивку по бизнес-направлениям

### 4.2 Roistat (Фаза 1)
- **Тип**: REST API
- **Тариф**: Бизнес 50 (полный доступ к API)
- **Проект**: #37488
- **ВАЖНО**: расходы в Roistat без НДС. Умножаем на 1.2 для реальных расходов
- **Каналы**: Яндекс.Директ 1 (основной), Яндекс.Директ 2, SEO, сарафан, колл-центр, Прямые визиты, Telegram, sotsignal, wazzup, marquiz, почта и др.
- **Кэш TTL**: 10 минут

### 4.3 Яндекс.Директ + Метрика (Фаза 2)
- **Тип**: REST API
- **2 аккаунта Директа** (основной + дополнительный)
- **Данные**: детальные расходы, показы, клики, конверсии на сайте

### 4.4 Rechka AI (Фаза 3)
- **Тип**: Google Sheets API
- **Источник**: сервис Rechka AI, ИИ прослушивает звонки менеджеров
- **Частота обновления**: 2–3 раза в день
- **Два листа**:
  - "Отчёт" — детализация каждого звонка (дата, менеджер, длительность, оценки по навыкам, обоснование, рекомендации)
  - "Дашборд" — агрегация по неделям (отдел + по менеджерам)
- **7 навыков** (оценка 0–100%): Установление контакта, Выявление потребностей, Усиление боли, Презентация, Отработка возражений, Предложение соответствующее потребностям, МОП лидер
- **Сводная оценка**: берётся КАК ЕСТЬ из Rechka AI, не пересчитывается
- **Кэш TTL**: 1 час

### 4.5 Adesk (Фаза 4)
- **Тип**: Webhooks (push-модель — Adesk сам отправляет данные)
- **Данные**: ДДС, ОПиУ, поступления, расходы по категориям
- **Счета**: ООО, ИП, наличные — все автоматически синхронизированы
- **ВАЖНО**: Adesk — основной источник финансовых данных, НЕ 1С
- **Кэш TTL**: 0 (real-time через webhooks)

### 4.6 1С Бухгалтерия (Фаза 4)
- **Версия**: Облачная (1С:Фреш)
- **Интеграция**: HTTP-сервис (делает 1С-программист по ТЗ)
- **Данные**: только дебиторская задолженность (кто должен, сколько, просрочка)
- **Кэш TTL**: 30 минут

### 4.7 1С УТ (Фаза 4)
- **Версия**: Локальная
- **Интеграция**: HTTP-сервис (делает 1С-программист по ТЗ)
- **Данные**: закупки оборудования, себестоимость, маржинальность
- **Связь с Битрикс24**: через поле "заказ клиента" → матчинг по имени контрагента (прямая синхронизация не работает, отказались)

- **Кэш TTL**: 30 минут

### 4.8 Claude API (Фаза 5)
- **Модель**: Claude Sonnet для AI-саммари
- **Данные на вход**: агрегированные метрики из всех источников

## 8. Бизнес-правила

1. **План продаж** считается по **валовому доходу** (не по выручке). Устанавливается РОПом поквартально, разбивается по месяцам. Хранится в таблице `sales_plans` в PostgreSQL.

2. **Время реакции на лид** вычисляется автоматически из Bitrix24: разница между датой создания лида и датой первого действия (звонок, письмо).

3. **Средний чек монтажа** исключает сделки ниже порога (по умолчанию 15 000 руб.) — чтобы продажи блока питания за 4000 руб. не портили статистику.

4. **Конверсии по стадиям** рассчитываются автоматически из исторических данных Битрикс24 (5137+ успешных сделок — отличная выборка).

5. **Расходы Roistat** умножаются на 1.2 (НДС 20%) для отображения реальных затрат.

6. **Дедупликация выездов**: если на одну сделку 2 осмотра — считаем 1. Если 2 монтажа — считаем 1. Ключ = deal_id. Повторные выезды видны в детализации, но не в агрегированной статистике.

7. **Закупки привязываются к сделкам** через matching по имени контрагента (1С УТ → Битрикс24). Закупки без привязки = складские.

8. **Оценки Rechka AI берутся as-is** — не пересчитываются, не переоценываются.

9. **Осмотры выполняют только монтажники**, не менеджеры.

10. **Контроль качества после монтажа** — планируются прозвоны. Когда запустятся, добавим как источник данных.

## 12. КРИТИЧНО: Логика дат и успешных сделок в Битрикс24

### Как определяется "успешная сделка за месяц"

Битрикс24 UI (фильтр "успех текущий месяц") использует комбинацию:
- **STAGE_ID = "C7:WON"** — сделка находится в стадии "Сделка успешна"
- **CLOSEDATE** (Дата завершения) — попадает в выбранный период

Наш дашборд повторяет эту логику: `Deal.is_won == True` + `Deal.closed_at` в нужном диапазоне.

### Поля дат в Битрикс24 (через REST API crm.deal.list)

| Поле API | Что означает | Используем? |
|----------|-------------|-------------|
| `DATE_CREATE` | Дата создания сделки | Да — для sync (загрузка новых сделок) |
| `CLOSEDATE` | **Дата завершения** (планируемая, но обновляется при закрытии) | **ДА — основное поле для метрик** |
| `DATE_MODIFY` | Дата последнего изменения (любое изменение) | Да — для sync (загрузка недавно изменённых) + "stale deals" |
| `STAGE_SEMANTIC_ID` | Семантика стадии: P=в работе, S=успех, F=провал | Да — определяет `is_won` / `is_lost` |
| `STAGE_ID` | Код стадии (C7:WON, C7:LOSE и т.д.) | Для справки |

### ВАЖНО: Особенности фильтрации API

1. **`CLOSEDATE` в API фильтруется через СТРОГОЕ неравенство**: `>CLOSEDATE` и `<CLOSEDATE` работают корректно, а `>=CLOSEDATE` возвращает ВСЕ записи (баг/особенность API). Но в нашей PostgreSQL это не проблема — SQLAlchemy работает с `closed_at` нормально.

2. **`STAGE_SEMANTIC_ID` — read-only в API**: нельзя фильтровать `crm.deal.list` по нему напрямую. Используем `STAGE_ID=C7:WON` для фильтрации выигранных, или загружаем все и проверяем `STAGE_SEMANTIC_ID == "S"` в коде.

3. **Sync загружает сделки двумя запросами**: по `DATE_CREATE` (новые) и по `DATE_MODIFY` (недавно изменённые). Это покрывает сценарий когда старая сделка (создана год назад) была только что переведена в "Сделка успешна".

### Что НЕ работает (проверено)

- **`crm.stagehistory.list`**: фильтр по `>=CREATED_TIME` НЕ работает в API (total всегда = полная история). Можно пагинировать и фильтровать вручную, но это медленно и ненадёжно. **Не используем.**
- **`DATE_MODIFY` как дата закрытия**: обновляется при ЛЮБОМ изменении сделки, не только при закрытии. Приводит к завышенным числам. **Не используем для метрик.**
- **Fallback `DATE_MODIFY` → `won_at`**: пробовали, даёт +300% завышение. **Отказались.**

### Правильный подход (текущая реализация)

```
Метрики "за месяц" = WHERE Deal.is_won == True AND Deal.closed_at >= начало_месяца
Метрики "вчера"    = WHERE Deal.is_won == True AND DATE(Deal.closed_at) == вчера
```

Это точно совпадает с фильтром "успех текущий месяц" в UI Битрикс24.

### Пока нет 1С

Метрика "План-Факт" на дашборде показывает **"Выручка (по сделкам)"** — это поле OPPORTUNITY из Битрикс24. Валовый доход (маржа) будет добавлен в Phase 4 после подключения 1С УТ.
